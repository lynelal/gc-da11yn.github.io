Standard on Information and Communication Technology Accessibility (ICT) Accessibility (Phase One)

Important information

  • Our soon to be awesome site is new and still under construction. Please open an issue on our Github project if you see anything that needs to be fixed or if you have any recommendations.

This page displays the Standard on Information and Communication Technologies (ICT) Accessibility in full, the various accessibility requirements and the associated implementation timelines

How to get involved?

To foster a participative engagement process, select stakeholders will be invited to:

  • Review the Standard on Information and Communication Technologies (ICT) Accessibility in full, the proposed accessibility requirements and the associated implementation timelines
  • Consult the background document with a description of each requirement
  • Submit your confidential and anonymous feedback by completing our feedback submission form

In the event of technical difficulties, stakeholders will be able to contact us through our generic mailbox: DigitalPolicy.PolitiqueNumerique-Consultation@tbs-sct.gc.ca

  1. 1. Effective date
    1. 1.1 This standard takes effect on April 1, 2023.
    2. 1.2 Building in accessibility from the start
      1. 1.2.1 Subsections 2.3.1.1, 2.3.2.1, 2.3.4.1 and 2.3.5.1 must conform with this standard by (effective date + 12 months)
    3. 1.3 Workforce capacity and capabilities
      1. 1.3.1 Subsections 2.3.8.1, 2.3.8.2 and 2.3.8.3 must conform with this standard by (effective date + 12 months)
    4. 1.4 Public-facing web content
      1. 1.4.1 Subsections 2.3.2.2 and 2.3.2.3 must conform with this standard by (effective date + 12 months)
    5. 1.5 Enterprise and departmental software
      1. 1.5.1 Subsection 2.3.4.2 must conform with this standard by (effective date + 12 months)
    6. 1.6 Native Mobile Applications (apps)
      1. 1.6.1 Subsection 2.3.5.2 must conform with this standard by (effective date + 24 months)
    7. 1.7 Internal-facing web content
      1. 1.7.1 Subsection 2.3.2.4 must conform with this standard by (effective date + 24 months)
    8. 1.8 User feedback mechanism
      1. 1.8.1 Subsection 2.3.7.1 must conform with this standard by (effective date + 12 months)
  2. 2. Standards
    1. 2.1 This standard provides details on the requirements set out in sections 4.4.1 and 4.4.2 of the Policy on Service and Digital and replaces the Standard on Web Accessibility dated August 1, 2011.
    2. 2.2 This standard applies to all information and communication technology (ICT) that is designed, procured, and used by the Government of Canada (GC).
    3. 2.3 Standards are as follows:
      1. 2.3.1 Procurement of ICT goods and services
        1. 2.3.1.1 Departments must meet the latest version of the Harmonised European Standard on Accessibility requirements for ICT products and services (EN 301 549) accessibility requirements when procuring new ICT products and services, while adhering to the Directive on the Management of Procurement;
      2. 2.3.2 Web content
        1. 2.3.2.1 All new web content must conform with the latest version of EN 301 549 (PDF, 2.3MB) and include an accessibility statement;
        2. 2.3.2.2 Accessibility conformance testing against the latest version of EN 301 549 standard must be conducted for all existing public-facing web content used by the department;
        3. 2.3.2.3 All existing public-facing web pages must feature an accessibility statement and a user feedback mechanism;
        4. 2.3.2.4 Accessibility conformance testing against the latest version of EN 301 549 standard must be conducted for all existing internal-facing web content used by the department.
      3. 2.3.3 Non-Web documents
        1. 2.3.3.1 Departments must provide an accessible version of any non-web documents when requested.
      4. 2.3.4 Enterprise and departmental software
        1. 2.3.4.1 All new enterprise and departmental software must conform with the latest version of EN 301 549 (PDF, 2.3MB) and include an accessibility statement;
        2. 2.3.4.2 All accessibility features built into all new applications must be configured to keep these features active when implemented in a department and must be available to users by default, while respecting security requirements in the Directive on Security Management;
      5. 2.3.5 Native mobile applications (apps)
        1. 2.3.5.1 All new native mobile applications must conform with the latest version of EN 301 549 (PDF, 2.3MB) and include an accessibility statement;
        2. 2.3.5.2 Accessibility conformance testing against the latest version of EN 301 549 standard must be conducted for all existing native mobile applications used by the department;
      6. 2.3.6 Hardware
        1. 2.3.6.1 Features that improve accessibility on all new hardware must be made available to users by default unless an exception has been granted.
      7. 2.3.7 User Feedback
        1. 2.3.7.1 Departments must provide mechanisms for users to give feedback on the accessibility of ICT.
      8. 2.3.8 Workforce capacity and capabilities
        1. 2.3.8.1 Training and tools for building in accessibility to ICT from the start must be available for public servants;
        2. 2.3.8.2 Departments must provide learning opportunities for all employees, including executives, on accessibility legislation, policies, and responsibilities for creating accessible ICT;
        3. 2.3.8.3 Departments must provide learning opportunities on accessibility requirements for employees who procure, develop or implement ICT.
      9. 2.3.9 Planning and performance reporting
        1. 2.3.9.1 Departments must ensure that their departmental plan for service and digital is informed by and aligned with the accessibility plan
        2. 2.3.9.2 Departments must report on the progress of ICT accessibility implementation in their departments according to guidance and accessibility measurement tools. These would be included in their accessibility plans and progress reports under the Accessible Canada Act.
(Note:** The definitions section is included to facilitate consultations and generate feedback but will be removed to respect TBS policy architecture. Definitions would be included in the Policy on Service and Digital and would be added in a future omnibus)

Appendix: Definitions

For the purpose of this standard, the following terms and definitions apply:

accessibility
the extent to which products, systems, services, environments and facilities can be used by people from a population with the widest range of user needs, characteristics and capabilities to achieve identified goals in identified contexts of use (Source: EN 301 549 (PDF, 2.3MB))
barrier
anything – including anything physical, architectural, technological or attitudinal, anything that is based on information or communications or anything that is the result of a policy or a practice – which hinders the full and equal participation in society of persons with an impairment, including a physical, mental, intellectual, cognitive, learning, communication or sensory impairment or a functional limitation (Source: Accessible Canada Act).
content
information and sensory experience to be communicated to the user by means of software, including code or mark-up, that defines the content’s structure, presentation and interactions (after WCAG2ICT [i.26]) (Source: EN 301 549 (PDF, 2.3MB)).
disability
any impairment, including a physical, mental, intellectual, cognitive, learning, communication or sensory impairment, or a functional limitation whether permanent, temporary or episodic in nature, or evident or not, that, in interaction with a barrier hinders a person’s full and equal participation in society (Source: Accessible Canada Act).
document
logically distinct assembly of content (such as a file, set of files, or streamed media) that functions as a single entity rather than as a collection. (Source: EN 301 549 (PDF, 2.3MB)).
information and communication technologies (ICT)
technology, equipment or interconnected system or subsystem of equipment for which the principal function is the creation, conversion, duplication, automatic acquisition, storage, analysis, evaluation, manipulation, management, movement, control, display, switching, interchange, transmission, reception or of data or information (Source: EN 301 549 (PDF, 2.3MB)).
non-web document
document that is not a web page, not embedded in web pages nor used in the rendering or functioning of the page (Source: EN 301 549 (PDF, 2.3MB)).
non-web software
software that is not a web page, not embedded in web pages nor used in the rendering or functioning of the page (Source: EN 301 549 (PDF, 2.3MB)).
user agent
a user agent is any software that retrieves and presents Web content for end users or is implemented using Web technologies (Source: World Wide Web Consortium (W3C)).
web content
content that belongs to a web page and that is used in the rendering or that is intended to be used in the rendering of the web page (Source: EN 301 549 (PDF, 2.3MB)).
web page
non-embedded resource obtained from a single URI (uniform resource identifier) using HTTP plus any other resources that are used in the rendering or intended to be rendered together with it by a user agent (Source: EN 301 549 (PDF, 2.3MB)).
